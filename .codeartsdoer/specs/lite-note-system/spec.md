# LiteNote轻量级全栈笔记系统 - 需求规格说明

## 1. 概述

### 1.1 系统简介
**LiteNote** 是一个专注于"快速记录"与"知识关联"的本地化笔记应用。本系统采用混合数据库架构，在无需外部服务依赖的情况下，通过Next.js实现功能完整、逻辑清晰、数据持久化的全栈应用。

### 1.2 技术栈概览
- **前端框架**: Next.js 14+ (App Router)
- **UI组件库**: Tailwind CSS + Shadcn/UI
- **关系型数据库**: SQLite (better-sqlite3) - 存储笔记元数据
- **文档数据库**: LowDB - 存储笔记正文内容
- **键值存储**: Keyv + @keyv/file - 存储缓存和统计数据

### 1.3 核心设计原则
- **本地优先**: 无需外部数据库服务，仅需Node.js运行时
- **混合架构**: SQLite元数据 + LowDB文档 + KV缓存
- **性能优先**: 利用KV存储实现高频读写优化
- **类型安全**: TypeScript全栈类型安全保障

## 2. 功能需求

### 2.1 笔记管理模块

#### 2.1.1 创建笔记
**需求描述**: 用户能够创建新的笔记，包含标题、正文内容和标签。

**验收标准**:
- **When** 用户在新建笔记页面输入标题、正文（支持Markdown）和标签（可选） **and** 点击保存按钮 **then** 系统应创建新笔记并跳转到详情页
- **When** 创建笔记成功 **then** SQLite应插入笔记元数据记录（id、title、tags、created_at、updated_at）
- **When** 创建笔记成功 **then** LowDB应在docs目录下创建对应的JSON文件存储正文内容
- **When** 创建笔记成功 **then** KV存储应初始化阅读计数为0（key: note:{id}:views）
- **When** 用户未输入标题 **then** 系统应显示错误提示并阻止提交
- **When** 标题超过200字符 **then** 系统应显示标题长度限制提示

#### 2.1.2 编辑笔记
**需求描述**: 用户能够修改已存在的笔记内容。

**验收标准**:
- **When** 用户在笔记详情页点击编辑按钮 **then** 系统应跳转到编辑页面并加载当前笔记内容
- **When** 用户修改标题或正文或标签 **and** 点击保存按钮 **then** 系统应更新SQLite元数据和LowDB文档内容
- **When** 编辑保存成功 **then** 系统应更新updated_at时间戳
- **When** 编辑保存成功 **then** 系统不应重置KV中的阅读计数
- **When** 编辑过程中发生错误 **then** 系统应显示错误提示并保留用户输入内容

#### 2.1.3 删除笔记
**需求描述**: 用户能够删除不再需要的笔记。

**验收标准**:
- **When** 用户在笔记详情页点击删除按钮 **then** 系统应显示确认删除对话框
- **When** 用户确认删除 **then** 系统应删除SQLite中的元数据记录
- **When** 用户确认删除 **then** 系统应删除LowDB中的JSON文档文件
- **When** 用户确认删除 **then** 系统应清理KV中相关的缓存数据（note:{id}:views）
- **When** 删除成功 **then** 系统应跳转到笔记列表页
- **When** 用户取消删除 **then** 系统应关闭对话框并保留笔记

#### 2.1.4 笔记列表展示
**需求描述**: 用户能够查看所有笔记的列表，并支持搜索和排序。

**验收标准**:
- **When** 用户访问首页 **then** 系统应显示所有笔记列表，按更新时间倒序排列
- **When** 笔记列表加载 **then** 每个笔记卡片应显示标题、摘要（前100字符）、标签、更新时间
- **When** 用户在搜索框输入关键词 **then** 系统应实时过滤标题包含该关键词的笔记
- **When** 搜索结果为空 **then** 系统应显示"未找到匹配笔记"提示
- **When** 笔记数量超过20条 **then** 系统应支持分页或无限滚动加载

---

### 2.2 内容详情与交互模块

#### 2.2.1 查看笔记详情
**需求描述**: 用户能够查看笔记的完整内容，支持Markdown渲染。

**验收标准**:
- **When** 用户点击笔记列表中的笔记卡片 **then** 系统应跳转到笔记详情页
- **When** 笔记详情页加载 **then** 系统应从LowDB读取JSON文档并渲染Markdown内容
- **When** Markdown内容包含代码块 **then** 系统应应用代码语法高亮
- **When** Markdown内容包含表格、列表、引用 **then** 系统应正确渲染相应格式
- **When** 笔记不存在 **then** 系统应显示404错误页面

#### 2.2.2 阅读量统计
**需求描述**: 系统自动统计每篇笔记的阅读次数。

**验收标准**:
- **When** 用户访问笔记详情页 **then** 系统应将KV中note:{id}:views的值加1
- **When** 笔记详情页显示 **then** 页面底部应显示当前阅读次数（如"128次阅读"）
- **When** 同一用户多次刷新页面 **then** 每次刷新都应增加阅读计数
- **When** KV中不存在该笔记的计数key **then** 系统应初始化为1

#### 2.2.3 热榜展示
**需求描述**: 在首页侧边栏展示阅读量最高的笔记列表。

**验收标准**:
- **When** 用户访问首页 **then** 侧边栏应显示"最近热门 Top 5"列表
- **When** 热榜加载 **then** 系统应从KV读取所有笔记的阅读计数并按降序排序
- **When** 热榜显示 **then** 每个热榜项应显示笔记标题和阅读次数
- **When** 用户点击热榜中的笔记 **then** 系统应跳转到对应笔记详情页
- **When** 笔记总数少于5条 **then** 热榜应显示所有现有笔记

---

### 2.3 标签与分类模块

#### 2.3.1 标签管理
**需求描述**: 用户能够为笔记添加标签，并按标签筛选笔记。

**验收标准**:
- **When** 用户创建或编辑笔记 **then** 可输入多个标签，以逗号分隔
- **When** 保存笔记 **then** 系统应将标签字符串存储到SQLite的tags字段
- **When** 笔记列表或详情页显示 **then** 应以标签徽章形式展示所有标签
- **When** 用户点击某个标签 **then** 系统应筛选并显示包含该标签的所有笔记
- **When** 标签筛选结果为空 **then** 系统应显示"该标签下暂无笔记"提示
- **When** 用户清除标签筛选 **then** 系统应恢复显示所有笔记

## 3. 非功能需求

### 3.1 性能需求
- **When** 用户打开笔记详情页 **then** 响应时间应小于50ms
- **When** 用户创建或编辑笔记 **then** 操作过程应无明显阻塞感
- **When** 笔记列表包含100条以上笔记 **then** 列表加载时间应小于200ms

### 3.2 可靠性需求
- **When** 系统重启 **then** 所有笔记数据、阅读量统计应保持持久化
- **When** 并发访问笔记详情页 **then** SQLite锁机制应保证数据一致性
- **When** 文件读写操作失败 **then** 系统应捕获异常并显示友好错误提示

### 3.3 易用性需求
- **When** 用户首次使用系统 **then** 界面应直观易懂，无需查看文档
- **When** 发生加载或操作 **then** 应显示适当的加载状态或进度提示
- **When** 操作成功或失败 **then** 应显示Toast通知反馈结果

### 3.4 兼容性需求
- **When** 用户使用移动设备访问 **then** 界面应自动适配，侧边栏转为抽屉菜单
- **When** 用户使用不同浏览器（Chrome、Firefox、Safari） **then** 系统应正常工作

## 4. 数据模型概览

### 4.1 SQLite元数据表 (notes)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (PK) | UUID格式唯一标识 |
| title | TEXT | 笔记标题 |
| tags | TEXT | 逗号分隔的标签字符串 |
| created_at | INTEGER | 创建时间戳 |
| updated_at | INTEGER | 更新时间戳 |

### 4.2 LowDB文档结构 (docs/{id}.json)
```json
{
  "id": "uuid-xxx",
  "content": "Markdown正文内容",
  "metadata": {
    "word_count": 120,
    "last_edited_by": "system",
    "custom_fields": {}
  }
}
```

### 4.3 KV存储结构
- `note:{id}:views` → Integer (阅读计数)
- `hot_notes:top5` → Array of IDs (可选，热榜缓存)

## 5. 约束与假设

### 5.1 技术约束
- 必须使用Next.js 14+ App Router架构
- 必须使用TypeScript确保类型安全
- 数据库操作必须通过Server Actions实现，避免客户端直接访问
- 所有数据文件必须存储在项目根目录的/data目录下

### 5.2 业务约束
- 单用户本地使用场景，无需用户认证系统
- 笔记标题最大长度200字符
- 标签最大长度50字符，单篇笔记最多10个标签

### 5.3 假设条件
- 用户具有基本的Markdown语法知识
- 运行环境已安装Node.js 18+
- 文件系统具有读写权限

---

## 6. 验收标准汇总

### 6.1 部署验收
- **When** 执行 `npm install && npm run dev` **then** 系统应成功启动，无需安装外部数据库服务

### 6.2 功能验收
- **When** 完成笔记的创建、编辑、删除、查看操作 **then** 所有功能应正常工作
- **When** Markdown内容包含各种格式 **then** 应正确渲染
- **When** 标签筛选功能使用 **then** 应准确过滤笔记

### 6.3 数据验收
- **When** 系统重启 **then** 笔记内容、标题、阅读量统计应完整保留

### 6.4 性能验收
- **When** 测量笔记详情页响应时间 **then** 应小于50ms
- **When** 创建笔记过程 **then** 应无阻塞感

## 7. 术语表

| 术语 | 定义 |
|------|------|
| SQLite | 轻量级关系型数据库，以单文件形式存储 |
| LowDB | 基于JSON文件的微型文档数据库 |
| KV存储 | 键值对存储，用于缓存和快速读写 |
| Server Actions | Next.js的服务端函数，用于数据变更操作 |
| Markdown | 轻量级标记语言，支持格式化文本 |
| UUID | 通用唯一标识符，用于生成笔记ID |
